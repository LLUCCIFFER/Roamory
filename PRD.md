# Roamory 产品需求文档

> 版本：v1.0 优化版  
> 日期：2026-06-24  
> 来源：基于 `AI_Travel_Memory_PRD.md` 分析、收敛和重组  
> 阶段：0-1 MVP 可开发版本  
> 产品定位：AI 旅行规划与旅行记忆资产管理工具

---

## 1. 产品结论

Roamory 不是普通攻略生成器，而是一个“从计划到回忆”的旅行生命周期产品。

一句话定位：

> 用 AI 帮用户生成可信、可编辑的旅行计划，并把每一次旅行沉淀为足迹、海报和可回看的记忆。

MVP 的核心闭环：

```text
输入目的地和偏好
-> 生成结构化行程
-> 用户编辑并保存
-> 生成分享海报
-> 点亮足迹城市
-> 下次继续规划或回看
```

第一版的目标不是把天气、相册、3D、社交全部做完，而是验证：

1. 用户是否愿意让 AI 生成旅行计划。
2. 用户是否认为生成结果可信、可改、可保存。
3. 用户是否愿意把保存的旅行变成个人足迹和分享素材。

## 3. 目标用户

### 3.1 周末短途旅行者

- 年龄：18-35 岁。
- 场景：周五晚上、周末临时想出门。
- 目标：快速找到 1-3 小时交通圈内适合拍照、吃饭、放松的去处。
- 价值：少查攻略，快速生成半日/一日/两日计划。

### 3.2 轻度旅行规划者

- 年龄：20-45 岁。
- 场景：节假日、年假、情侣游、朋友游、亲子游。
- 目标：解决景点顺序、交通时间、预算、一天能玩多少的问题。
- 价值：生成一份可编辑、可保存、可分享的行程。

### 3.3 旅行记录爱好者

- 年龄：18-40 岁。
- 场景：喜欢地图、照片、人生清单、纪念品。
- 目标：知道自己去过哪里，并把旅行变成长期记忆。
- 价值：足迹地图、旅行卡片、年度报告、纪念收藏。

## 4. 产品原则

1. AI 只做编排和表达，地点、路线、天气、照片元数据必须来自可校验数据源。
2. 第一版必须让用户能控制结果：删除、替换、重排、保存。
3. 默认保护隐私：行程默认私密，分享前必须展示公开内容。
4. 先做可用闭环，再做差异化探索。
5. 降级体验必须明确：地图、AI、海报、网络失败时不阻塞已保存内容查看。

---

## 4.1 已确认启动决策

| 决策项 | 结论 |
|---|---|
| 首发形态 | 多端。产品、设计和接口按 Web / 移动端 / 小程序可复用的方式设计。 |
| 地图服务商 | 高德地图。POI、路线、天气等地图相关能力优先围绕高德 Adapter 实现。 |
| AI 服务 | 采用可替换 LLM Adapter；具体模型供应商通过配置接入，不写死在业务逻辑中。 |
| 结构化输出 | TripPlan JSON Schema + 服务端 Schema 校验 + 一次自动修复重试 + 失败降级。 |
| 账号策略 | MVP 先游客 + 本地存储；保存、同步或公开分享时再引导登录/绑定。 |
| 分享策略 | MVP 同步支持公开链接，并默认走隐私检查。 |
| 相册记忆 | P1 允许上传缩略图；原图默认不上传，需用户单独确认。 |

## 7. 核心流程

### 7.1 创建并保存 AI 行程

```text
进入规划页
-> 输入目的地、天数、预算、兴趣
-> 选择出发城市、日期、同行人、节奏、交通方式
-> 点击“生成我的行程”
-> 系统创建 generationTask
-> 展示生成进度
-> 生成 Trip 草稿
-> 用户查看、编辑、保存
```

成功标准：

- 生成结果包含完整天数。
- 每天有 2-5 个主要 POI，随节奏变化。
- POI 包含名称、类型、坐标、推荐理由。
- 连续点位之间有通勤时间或“待确认”标记。
- 用户可以保存为正式 Trip。

### 7.2 编辑行程

```text
进入行程结果页
-> 切换 Day
-> 删除、替换、重排或锁定点位
-> 系统重新计算时间轴和路线
-> 用户确认保存
```

成功标准：

- 删除点位后时间轴更新。
- 拖拽排序后通勤信息刷新。
- 锁定点位不会被后续 AI 自动替换。
- 路线 API 失败时，行程仍可保存。

### 7.3 点亮足迹

```text
保存行程
-> 系统识别目的地城市
-> 询问是否点亮该城市
-> 用户确认
-> 城市加入足迹
-> 城市详情关联 Trip
```

成功标准：

- 用户可以手动搜索并点亮城市。
- 保存行程后可以点亮关联城市。
- 城市详情展示旅行记录数、最近到访、关联 Trip。
- 私密城市不会出现在公开分享中。

### 7.4 分享海报与隐私检查

```text
进入分享海报页
-> 选择模板和展示内容
-> 系统展示公开内容检查
-> 用户确认
-> 生成图片或分享链接
```

成功标准：

- 用户能隐藏预算、酒店区域、精确位置。
- 生成海报清晰，无内容错位。
- 私密行程不可生成公开链接，或必须改为 link_only。
- 删除分享链接后公开页立即不可访问。

## 8. 功能需求

### FR-001 游客模式与登录

优先级：P0

需求：

- 未登录用户可进入首页并生成一次行程草稿。
- MVP 先使用游客 + 本地存储保存草稿和本机行程。
- 保存到云端、跨设备同步、生成公开链接时需要登录或绑定账号。
- 登录方式优先邮箱/手机号，第三方登录可后置。

验收：

- 未登录可生成并预览。
- 保存时弹出登录引导。
- 登录后草稿可绑定到用户账号。
- 用户可注销、导出、删除个人数据。

### FR-002 旅行偏好

优先级：P0

字段：

| 字段 | 类型 |
|---|---|
| interests | string[] |
| travelPace | relaxed / balanced / intense |
| budgetLevel | low / medium / high |
| transportPreference | walking / public_transport / driving / taxi |
| companionType | solo / couple / friends / family / elder / child |
| avoidTags | string[] |
| defaultDepartureCity | string |

验收：

- 用户可新增、修改、删除偏好。
- 创建行程时自动带入默认偏好。
- 本次调整可选择“仅本次使用”或“保存为默认”。

### FR-010 AI 行程生成

优先级：P0

输入：

| 字段 | 必填 | 说明 |
|---|---|---|
| destination | 是 | 目的地城市或区域 |
| days | 是 | 旅行天数 |
| budgetAmount | 否 | 总预算 |
| interests | 否 | 兴趣偏好 |
| departureCity | 否 | 出发城市 |
| startDate | 否 | 出行日期 |
| companionType | 否 | 同行人 |
| travelPace | 否 | 节奏 |
| transportMode | 否 | 交通方式 |
| mustVisitPois | 否 | 必去点 |
| avoidPois | 否 | 避开点 |
| freeText | 否 | 自然语言补充 |

输出：

- 行程标题、目的地、天数。
- 每日主题、时间轴、POI、餐饮建议。
- POI 名称、坐标、类型、推荐理由、预计停留时间。
- 点位间通勤方式、时间、距离。
- 每日预算估算。
- 雨天备选。
- 风险提示。
- 可行性评分。

规则：

- 禁止编造不存在的地点。
- POI 必须来自地图/内部数据源或用户输入。
- 信息缺失时用“待确认”，不能伪造。
- 每日总活动强度不超过 10 小时。
- 生成失败必须给出错误原因和重试建议。

### FR-011 行程局部修改

优先级：P0

支持：

- 重新生成全部。
- 重新生成某一天。
- 替换某个点位。
- 删除某个点位。
- 插入新点位。
- 降低强度。
- 生成雨天方案。

验收：

- 局部修改不要求用户重填全部表单。
- 用户锁定点位不被替换。
- 修改后时间轴和通勤信息更新。

### FR-020 行程地图/列表编辑

优先级：P0

MVP 可先使用“地图缩略图 + 时间轴列表”，真实路线编辑可在后续轮次增强。

必须能力：

- 展示 POI 顺序。
- 展示通勤时间或待确认。
- 拖拽排序。
- 删除、替换、锁定。
- 保存修改。

验收：

- 顺序调整后刷新仍保留。
- 网络失败时展示列表版行程。
- 地图 API 失败不影响保存。

### FR-030 分享海报

优先级：P0

MVP 模板：

- 行程卡片。
- 每日时间轴长图。
- 足迹城市卡片。

内容开关：

- 显示/隐藏预算。
- 显示/隐藏具体时间。
- 显示/隐藏精确位置。
- 显示/隐藏地图。

验收：

- 图片生成成功率可监控。
- 隐藏项不会出现在图片和公开链接中。
- 海报生成失败可重试。
- MVP 支持生成公开链接，链接使用不可猜测 token。
- 公开链接默认只展示用户确认过的内容。

### FR-040 足迹地图

优先级：P0

MVP 能力：

- 手动点亮城市。
- 保存行程后点亮城市。
- 城市详情关联 Trip。
- 私密城市控制。

验收：

- 用户能搜索城市并点亮。
- 点亮统计实时更新。
- 删除最后一条关联 Trip 时询问是否取消点亮。

### FR-080 隐私与数据管理

优先级：P0

默认值：

| 数据 | 默认 |
|---|---|
| 行程 | private |
| 预算 | 不分享 |
| 酒店精确位置 | 不分享 |
| 精确轨迹 | 城市级模糊 |
| 原图上传 | 关闭 |
| 相册扫描 | 主动授权 |
| 缩略图上传 | P1 可开启，需用户确认 |

验收：

- 触发权限前展示用途说明。
- 分享前展示公开内容清单。
- 用户可以删除 Trip、Footprint、SharePoster。
- 日志不得记录精确照片坐标和原图内容。

## 11. AI 编排原则

AI 生成流程：

```text
用户输入
-> 意图解析
-> POI 检索
-> 候选 POI 评分
-> 路线矩阵计算
-> 规则排程
-> LLM 生成解释文案
-> JSON Schema 校验
-> 保存为 Trip 草稿
```

AI 约束：

- 输出必须是 JSON。
- 结构化输出采用 TripPlan JSON Schema。
- LLM 接入通过可替换 Adapter 封装，具体模型供应商通过环境配置指定。
- 不允许输出没有来源的 POI。
- 不允许伪造坐标、营业时间、价格。
- 信息缺失必须标记“待确认”。
- JSON Schema 校验失败时自动重试一次，仍失败则返回错误。

POI 评分建议：

```text
poiScore =
  interestMatchScore * 0.35
+ popularityScore * 0.20
+ distanceScore * 0.20
+ budgetFitScore * 0.10
+ weatherFitScore * 0.05
+ noveltyScore * 0.10
```

## 12. 成功指标

北极星指标：

```text
每月每个活跃用户保存的有效旅行记录数
```

有效旅行记录定义：

- 至少 1 个目的地城市。
- 至少 1 天行程，或 P1 阶段至少 3 张照片。
- 用户主动保存，或用户确认由相册生成。
- 24 小时内未被删除。

MVP 上线 4-8 周目标：

| 指标 | 目标 |
|---|---:|
| 首次行程生成完成率 | >= 60% |
| 行程保存率 | >= 35% |
| 分享海报生成率 | >= 20% |
| 足迹点亮率 | >= 30% |
| 7 日留存 | >= 15% |
| 用户行程满意评分 | >= 4.0 / 5 |
| 第二次行程创建率 | >= 15% |

## 13. 非功能需求

| 场景 | 要求 |
|---|---|
| 首屏加载 | 2 秒内展示缓存或骨架屏 |
| AI 生成 | 异步任务，前端显示步骤和进度 |
| 拖拽排序 | UI 响应小于 300ms |
| 路线重算 | 尽量 5 秒内返回，失败可降级 |
| 海报生成 | 10 秒内成功或返回可重试状态 |
| 无网络 | 已保存行程可只读查看 |

安全与合规：

- 全接口 HTTPS。
- 用户数据按 userId 隔离。
- 分享链接使用不可猜 token。
- 私密行程不可公开访问。
- 对象存储使用签名 URL 或权限控制。
- 日志不记录敏感坐标、原图内容、完整轨迹。

## 14. 风险与对策

| 风险 | 严重度 | 对策 |
|---|---|---|
| 范围过大 | 高 | MVP 只做计划、保存、分享、足迹。 |
| AI 编造地点 | 高 | POI 检索前置，LLM 只能基于候选数据生成。 |
| 地图 API 成本高 | 中 | 缓存路线，MVP 支持待确认降级。 |
| 隐私泄露 | 高 | 默认私密，分享前强制检查公开内容。 |
| 海报生成质量不稳定 | 中 | 先做少量模板，逐个验收。 |
| 相册权限转化低 | 中 | P1 再做，并强化本地优先说明。 |
